TUTORIAL 01

What Are Copilot Custom Agents?

10 minBeginner

Walkthrough

Step 1: The Three File Types

GitHub Copilot custom agents are built from three file types living in your repo:

TypeLocationPurpose
Agent.github/agents/WHAT to do (workflow, orchestration)
Tree.copilot_utils/context/trees/HOW to decide (if/else logic)
Knowledge.copilot_utils/context/knowledge/WHAT to use (reference info, patterns)

Step 2: Directory Layout

Project Structure
.github/agents/                    # Agent workflow files
    └── my-agent.agent.md
.copilot_utils/
    └── context/
        ├── trees/                 # Decision trees (if/else)
        │   ├── strategy_selection.md
        │   └── flag_selection.md
        └── knowledge/             # Reference knowledge
            ├── anti_patterns.md
            └── code_patterns.md

Step 3: The Separation Principle

  • Agents contain workflow only — never embed knowledge or decision logic
  • Trees contain branching decisions — one file per decision type
  • Knowledge contains reference info — many small files, not one big file
Key Rule: Agents reference trees and knowledge files — they never embed them. This keeps agents focused on orchestration and makes knowledge reusable across agents.

Key Takeaways

  • Three file types: Agent (workflow), Tree (decisions), Knowledge (reference)
  • Agents live in .github/agents/, context in .copilot_utils/context/
  • Separation principle: agents reference, never embed
TUTORIAL 02

Your First Agent — Frontmatter & Body

15 minBeginner

Walkthrough

Step 1: Minimal Agent

.github/agents/my-reviewer.agent.md
---
name: My-Reviewer
description: 'Reviews code for anti-patterns and suggests fixes'
tools: ['read', 'edit', 'search']
model: 'Claude Sonnet 4.5'
---

## Role

You are a senior code reviewer focused on identifying
anti-patterns and suggesting improvements.

## Dynamic Parameters

- **filePath**: Path to the file to review

## Variable Extraction Strategy

1. **From User Input**: Parse file path from argument
2. **From File Context**: Use current open file
3. **Ask User**: If not provided, request it

## Workflow

### PHASE 1: Read and Analyze

1. Read the file at `${filePath}`
2. Identify anti-patterns and issues
3. Categorize by severity: CRITICAL, HIGH, MEDIUM

### PHASE 2: Generate Report

1. List all findings with line numbers
2. Suggest fix for each issue
3. Provide overall quality score

## Return Format

**Status:** PASS | FAIL
**Issues Found:** count
**Report:** markdown table of findings

Step 2: Frontmatter Properties

PropertyRequiredDescription
nameOptionalDefaults to filename
descriptionYes50-150 chars, single-quoted
toolsOptionalOmit = all tools. ['read', 'edit', 'search', 'execute', 'agent', 'web', 'todo']
modelRecommended'Claude Sonnet 4.5', 'gpt-4o'
targetOptional'vscode' or 'github-copilot'
inferOptionaltrue = auto-invoke by context
argument-hintOptionalInput hint (VS Code only)
handoffsOptionalSub-agent delegations (VS Code only)

Step 3: Tool Aliases

Available Tools
# Standard tools
tools: ['read', 'edit', 'search', 'execute', 'agent', 'web', 'todo']

# MCP server tools
tools: ['github/*']                    # All GitHub tools
tools: ['github/get_file_contents']    # Specific tool
tools: ['playwright/*']                # Browser automation
tools: ['my-mcp/tool-name']           # Custom MCP server

Step 4: Agent Body Sections

SectionPurpose
RoleAgent's expertise and focus area
Dynamic ParametersExpected input variables
Variable Extraction StrategyHow to get variables (input, context, ask)
WorkflowMulti-phase execution (PHASE 1, PHASE 2...)
Return FormatStructured output definition

Step 5: Naming Convention

Agent files use lowercase-with-hyphens.agent.md. The .agent.md suffix tells VS Code this is an agent file.

Try It Yourself

Create .github/agents/hello-world.agent.md with a description, read tool only, and a single-phase workflow that reads a file and summarizes it. Invoke it in VS Code Copilot Chat with @hello-world.

Hint

Keep it minimal: one tool (read), one phase, one output. The description should be under 150 chars. Test with @hello-world summarize this file.

Key Takeaways

  • Agents are .agent.md files in .github/agents/
  • Frontmatter: description is required, tools and model recommended
  • Body sections: Role, Dynamic Parameters, Variable Extraction, Workflow, Return Format
  • Invoke with @agent-name in Copilot Chat
TUTORIAL 03

Context Files — Trees & Knowledge

15 minBeginner

Walkthrough

Step 1: Decision Trees

Trees contain if/else logic for agent decisions:

.copilot_utils/context/trees/environment_selection.md
# Environment Selection Tree

Input: `{environment}` parameter

Decision:
├── "development" → Use local config, enable debug logging
├── "staging"     → Use staging API, enable monitoring
├── "production"  → Use prod config, enable rate limiting
└── Unknown       → Default to "development"

Step 2: Knowledge Files

Knowledge files contain reference information — patterns, rules, examples:

.copilot_utils/context/knowledge/anti_patterns.md
# Anti-Patterns to Identify

## Vague Verbs
- "handle" → too ambiguous, specify exact action
- "process" → what processing? be specific
- "manage" → manage how? define the operation

## Combined Actions
- One function doing multiple unrelated things
- Single endpoint handling different resource types
- Mixed concerns in same module

## Missing Error Handling
- No try/catch around I/O operations
- Silently swallowing exceptions
- Generic error messages without context

Step 3: Referencing in Agents

In your agent's workflow
### PHASE 2: Apply Decision

**Apply tree:** `.copilot_utils/context/trees/environment_selection.md`

1. Evaluate `{environment}` against tree criteria
2. Set `configPath` based on match

### PHASE 3: Validate

**Reference:** `.copilot_utils/context/knowledge/anti_patterns.md`

1. Check code against each pattern listed
2. Report any matches found

Step 4: Decomposition Principle

File Organization
# BAD — one large file
knowledge/
└── everything.md (500 lines)

# GOOD — many small files
knowledge/
├── anti_patterns.md (50 lines)
├── setup_patterns.md (40 lines)
├── teardown_patterns.md (30 lines)
└── validation_rules.md (45 lines)

# Same for trees — one per decision
trees/
├── environment_selection.md
├── strategy_selection.md
└── output_format_selection.md

Key Takeaways

  • Trees = decision logic (if/else). Reference with **Apply tree:**
  • Knowledge = reference info (patterns, rules). Reference with **Reference:**
  • Many small files beat one large file — decompose aggressively
  • Agents reference context files, never embed their content

Walkthrough

Phase Pattern

**Before starting:** Mark todo #1 as "in-progress"

2. Identify file type and structure
3. Prepare context for next phase

Call `manage_todo_list` with:


**During Execution:**
- Mark task as `"in-progress"` when starting
- Mark task as `"completed"` after finishing

Step 3: Variable Chaining Between Phases

Phase-to-Phase Data Flow
### PHASE 1: Extract Data

**Input:** User provides `{inputId}`
**Output Variables:**
- `inputId` = "12345"
- `inputType` = "type_a"

---

### PHASE 2: Apply Decision

**Input:** Uses `inputType` from Phase 1
**Apply tree:** `.copilot_utils/context/trees/tree_1.md`
**Output Variables:**
- `outputPath` = "path/to/output"

---

### PHASE 3: Generate Output

**Input:** Uses `outputPath` and `inputId` from previous phases
**Steps:**
1. Construct file path: `${outputPath}/${inputId}.py`
2. Generate content using template

PatternExample
Declaration**paramName**: Description in Dynamic Parameters

  • Each phase has: Before starting, Steps, Output Variables, After completion
  • Todo tracking with manage_todo_list shows progress in VS Code sidebar
  • Variables flow phase-to-phase via Output Variables declarations
  • Use ${variableName} syntax with camelCase naming
TUTORIAL 05

Sub-Agent Orchestration

20 minIntermediate

Walkthrough

Step 1: The Orchestrator Pattern

Orchestrator Flow
Orchestrator → (FILE_PATH + expectations) → Sub-Agent
Sub-Agent    → (structured JSON result)    → Orchestrator

The orchestrator never does the work itself. It gathers context, delegates to specialists, and coordinates results.

Step 2: VS Code — handoffs Syntax

.github/agents/orchestrator.agent.md
---
name: VS-Code-Orchestrator
description: 'Orchestrates validation workflow with sub-agents'
tools: ['read', 'edit', 'search', 'agent', 'todo']
model: 'Claude Sonnet 4.5'
target: 'vscode'
argument-hint: 'Provide file path to process'
handoffs:
    - label: 'Validate File'
      agent: 'Example-Validator'
      prompt: 'Analyze this file for anti-patterns and return violations'
    - label: 'Transform File'
      agent: 'Example-Transformer'
      prompt: 'Apply fixes for the violations found'
---

## Workflow

### PHASE 2: Delegate Analysis

**Delegate to:** `Example-Validator` via handoff

**Input:**
- File: `${filePath}`
- Rules: `.copilot_utils/context/knowledge/anti_patterns.md`

**Expected return:**
```json
{ "status": "PASS | FAIL", "violations": [...], "count": 0 }
```

**On PASS:** Continue to Phase 3
**On FAIL:** Record violations, continue for fixes

Step 3: GitHub Copilot — runSubagent Syntax

runSubagent() Pattern
// Phase 1: Analyze
const analysisResult = await runSubagent({
  description: 'Analyze file for patterns',
  prompt: `You are the Analyzer.

Input:
- File: ${filePath}
- Reference: .copilot_utils/context/knowledge/anti_patterns.md

Task:
1. Read the file
2. Identify all anti-pattern matches
3. Return findings

Return JSON:
{ "patterns": ["..."], "lineNumbers": [10, 25], "suggestions": ["..."] }`
});

// Phase 2: Transform (chains result from Phase 1)
const transformResult = await runSubagent({
  description: 'Apply transformations',
  prompt: `You are the Transformer.

Input from Analyzer:
- Patterns: ${analysisResult.patterns}
- Lines: ${analysisResult.lineNumbers}

Task:
1. Apply fixes at specified lines
2. Write updated file

Return JSON:
{ "status": "success", "changesApplied": 3 }`
});

Step 4: Orchestration Rules

  • Orchestrator provides context — file path + expectations for each sub-agent
  • Sub-agents return structured data — JSON format back to orchestrator
  • Parent tools limit child — sub-agents can't use tools the parent doesn't have
  • Keep sub-agents focused — one clear task per sub-agent
  • Chain results — pass output from one sub-agent as input to next
  • Limit scale — max 5-10 sequential sub-agent calls

Key Takeaways

  • VS Code uses handoffs in frontmatter for sub-agent delegation
  • GitHub Copilot uses runSubagent() for inline sub-agent calls
  • Orchestrator delegates, never does the work itself
  • Sub-agents return structured JSON, results chain between phases
TUTORIAL 06

Prompt Chaining — Multi-Step Pipelines

20 minIntermediate

Walkthrough

Step 1: What Is a Prompt Chain?

A prompt chain is a sequence of prompts that execute in order, each building on the outputs of previous steps. Two file types:

TypePurposeExample
Magic PromptOrchestrator — sets params, defines ordermagic-prompt.md
Sub-PromptSingle step — does one task1-analyze.md

Step 2: Magic Prompt Template

.github/prompts/magic-prompt.md
{output_folder} = .results
{final_output_file} = ./output/final.md

You are executing a multi-step prompt chain.

1. Navigate to the prompts folder
2. Review all prompt files WITHOUT executing them
3. Confirm you understand the full chain sequence
4. Execute prompts in numerical order:
    - 1-analyze-structure.md
    - 2-document-components.md
    - 3-generate-docs.md
5. For each step, output results to `{output_folder}/`

Stop ONLY when:
- All steps are complete
- A full `{final_output_file}` can be generated

Step 3: Sub-Prompt Template

.github/prompts/1-analyze-structure.md
You are a senior developer analyzing codebase structure.

Your task:

1. Identify all directories and their purposes
2. List key files in each directory
3. Output to ./{output_folder}/1-structure.md

This task may take some time — that is expected and acceptable.
Do not skip files or produce partial results.
Accuracy and completeness are mission-critical.

After writing ./{output_folder}/1-structure.md, read the contents
of ./2-document-components.md and proceed accordingly
with {output_folder} as the output-folder.

Step 4: Variable Syntax

Variables
# Declaration (in magic prompt)
{output_folder} = .results
{final_output_file} = ./output/final.md

# Usage (in sub-prompts)
./{output_folder}/1-analysis.md

# Chain command (link to next step)
After writing ..., read ./2-next.md and proceed with {output_folder}

Step 5: Quality Guardrails

Include in every sub-prompt to prevent shortcuts:

Guardrail Block
This task may take some time — that is expected and acceptable.
Do not skip files or produce partial results due to time or complexity.
Accuracy and completeness are mission-critical.
Do not optimize for speed or brevity.

Key Takeaways

  • Magic prompt = orchestrator. Sub-prompts = individual steps.
  • Variables declared with {name} = value, used with {name}
  • Chain command links steps: "After writing..., read next and proceed"
  • Quality guardrails prevent the model from cutting corners

VS Code vs GitHub.com — Differences

15 minAdvanced

Walkthrough

Step 1: Feature Comparison

FeatureVS CodeGitHub.com
model propertyYesNo
argument-hintYesNo
handoffsYesNo
todo toolYesNo
runSubagentYesYes
github/* toolsLimitedFull access
Custom MCP in agentNoOrg/Enterprise only
File editingInteractive previewDirect edits

Step 2: Target-Specific Agents

VS Code Only
---
name: VS-Code-Agent
target: 'vscode'
tools: ['read', 'edit', 'todo']   # todo only works in VS Code
handoffs:
    - label: 'Validate'
      agent: 'Validator'
      prompt: 'Run validation checks'
---
GitHub.com Only
---
name: GitHub-Agent
target: 'github-copilot'
tools: ['read', 'edit', 'github/*']   # Full GitHub API access
---

Step 3: MCP Server Configuration

MCP in Agent (Org/Enterprise only)
---
tools: ['read', 'my-server/*']
mcp-servers:
    my-server:
        type: 'local'
        command: 'npx'
        args: ['-y', '@org/mcp-server']
        env:
            API_KEY: ${{ secrets.API_KEY }}
---

Key Takeaways

  • VS Code has handoffs, todo, argument-hint, model — GitHub.com does not
  • runSubagent() works in both environments
  • Use target: to restrict agents to a specific environment
  • Custom MCP servers in agents are Org/Enterprise only on GitHub.com

Step 2: Complete Orchestrator Example

.github/agents/code-processor.agent.md
---
name: Code-Processor
description: 'Analyzes code for anti-patterns and applies fixes via sub-agents'
tools: ['read', 'edit', 'search', 'agent', 'todo']
model: 'Claude Sonnet 4.5'
target: 'vscode'
argument-hint: 'Provide file path to process'
handoffs:
    - label: 'Validate File'
      agent: 'Example-Validator'
      prompt: 'Analyze for anti-patterns using knowledge_1.md rules'
    - label: 'Transform File'
      agent: 'Example-Transformer'
      prompt: 'Apply fixes for violations found'
---

## Role
You are a senior engineer orchestrating a code processing workflow.
You delegate work to specialists and coordinate their results.

## Dynamic Parameters
- **filePath**: Path to the file to process

## Variable Extraction Strategy
1. **From User Input**: Parse file path from argument
2. **From File Context**: Use current open file
3. **Ask User**: If not provided, request it

## PROGRESS TRACKING
Call `manage_todo_list` with:
```json
{
  "operation": "write",
  "todoList": [
    { "id": 1, "title": "Gather context", "status": "not-started" },
    { "id": 2, "title": "Delegate analysis", "status": "not-started" },
    { "id": 3, "title": "Delegate fixes", "status": "not-started" },
    { "id": 4, "title": "Compile report", "status": "not-started" }
  ]
}
```

## Workflow

### PHASE 1: Gather Context
**Before starting:** Mark todo #1 as "in-progress"
1. Read the file at `${filePath}`
2. Identify file type and structure
**Output Variables:**
- `fileType` = detected type
- `fileContent` = content summary
**After completion:** Mark todo #1 as "completed"

### PHASE 2: Delegate Analysis
**Before starting:** Mark todo #2 as "in-progress"
**Delegate to:** `Example-Validator` via handoff
**Input:**
- File: `${filePath}`
- Reference: `.copilot_utils/context/knowledge/anti_patterns.md`
**Expected return:**
```json
{ "status": "PASS | FAIL", "violations": [...], "count": 0 }
```
**After completion:** Mark todo #2 as "completed"

### PHASE 3: Delegate Fixes
**Before starting:** Mark todo #3 as "in-progress"
**Delegate to:** `Example-Transformer` via handoff
**Input:**
- File: `${filePath}`
- Violations: `${analysisResult.violations}`
- Reference: `.copilot_utils/context/knowledge/code_patterns.md`
**After completion:** Mark todo #3 as "completed"

### PHASE 4: Compile Report
**Before starting:** Mark todo #4 as "in-progress"
1. Collect results from both sub-agents
2. Generate summary with findings and fixes applied
**After completion:** Mark todo #4 as "completed"

## Error Handling
- **Sub-agent timeout**: Retry once, then report failure
- **Invalid file path**: Ask user for correct path

## Return Format
**Status:** success | failed
**Analysis:** PASS | FAIL (violation count)
**Fixes Applied:** count
**Output:** Final file path
