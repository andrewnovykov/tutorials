TUTORIAL 01

What Is Claude Code?

15 min Beginner

What You'll Learn

  • What Claude Code is and how it works
  • The .claude/ folder system: agents, commands, skills, hooks
  • Why structure beats raw prompting
  • The building blocks that turn Claude into an engineering system

Context

Claude Code is Anthropic's CLI for Claude — an AI coding assistant that runs in your terminal. Out of the box it's powerful, but raw prompting without structure leads to inconsistent results. The real power comes from the .claude/ folder — a convention-based system of agents, commands, skills, and hooks that transforms Claude from a chatbot into a repeatable engineering system.

Think of it this way: Claude Code is the engine, the .claude/ folder is the transmission, steering, and instruments that let you actually drive.

Walkthrough

Step 1: The Five Pillars

Claude Code's extension system is organized around five core concepts:

PillarWhat It DoesLocation
AgentsSpecialized AI workers with scoped tools and models.claude/agents/*.md
CommandsSlash commands that trigger workflows.claude/commands/*.md
SkillsDomain knowledge packages loaded on demand.claude/skills/*/SKILL.md
HooksEvent-driven automations before/after tool callssettings.json
CLAUDE.mdProject brain — always-on rules and contextCLAUDE.md

Step 2: The Architecture

Here's how a typical Claude Code project is organized:

Project Structure
your-project/
├── .claude/
│   ├── agents/          # Specialized AI subagents
│   ├── commands/        # Slash commands (workflows)
│   ├── skills/          # Domain knowledge packages
│   │   └── my-skill/
│   │       ├── SKILL.md       # Core knowledge
│   │       ├── FORMAT.md      # Output format spec
│   │       ├── VOICE.md       # Tone and terminology
│   │       ├── workflows/     # Task playbooks
│   │       └── references/    # On-demand docs
│   ├── workflows/       # Multi-agent pipeline definitions
│   ├── settings.json    # Hooks, permissions, MCP servers
│   └── mcp.json         # MCP server configurations
├── CLAUDE.md            # Project brain (always loaded)
└── src/                 # Your actual code

Step 3: Why Structure Matters

Without structure, every Claude session starts from scratch. You tell it your standards, hope it remembers your patterns, and manually catch mistakes. With the .claude/ system:

  • Agents handle planning, reviewing, and testing autonomously
  • Hooks catch security violations before code is committed
  • Skills inject the right domain knowledge at the right time
  • CLAUDE.md enforces your standards without repeating yourself

Step 4: The Key Principle — Roles, Not Hierarchy

Every component in the .claude/ system is a role, not a hierarchy level. Commands, skills, agents — they're all callable from anywhere:

  • Commands — reusable operations, callable by humans AND agents
  • Skills — knowledge packages any agent can load
  • Agents — autonomous workers spawned via the Agent tool
  • MCP Servers — capability extensions shared by all components
Cross-Tool Insight: These concepts transfer to other AI coding tools. Cursor uses .cursorrules. Codex uses AGENTS.md and .codex/config.toml. The patterns you learn here apply across all AI tools.

Key Takeaways

  • Claude Code + the .claude/ folder = a structured engineering system
  • Five pillars: agents, commands, skills, hooks, CLAUDE.md
  • Every component is a role, not a hierarchy — callable from anywhere
  • Structure beats raw prompting — repeatable workflows produce better results
TUTORIAL 02

Getting Started — Setup Your Project

20 min Beginner

What You'll Learn

  • How to install Claude Code CLI
  • How to create the .claude/ folder structure
  • How to write your first agent, command, and skill
  • How to verify everything works

Context

Setting up Claude Code is straightforward. The CLI installs globally, and the .claude/ folder lives inside your project. No build tools, no compilation — just markdown files that Claude reads automatically.

Walkthrough

Step 1: Install Claude Code

Terminal
# Install Claude Code CLI globally
npm install -g @anthropic-ai/claude-code

# Verify installation
claude --version

# Start Claude Code in your project
cd your-project
claude

Step 2: Create the .claude/ Folder Structure

Terminal
# Create the core directory structure
mkdir -p .claude/agents
mkdir -p .claude/commands
mkdir -p .claude/skills
mkdir -p .claude/workflows

# Create your project brain
touch CLAUDE.md

Step 3: Create Your First CLAUDE.md

CLAUDE.md
# My Project

## Overview
Brief description of what this project does and tech stack.

## Build & Test
npm install      # Install dependencies
npm test         # Run tests
npm run build    # Build the project

## Conventions
- Use TypeScript strict mode
- Prefer functional patterns
- Error handling: always wrap with context
- Tests: table-driven with descriptive names

Step 4: Verify It Works

Terminal
# Start Claude Code — it will auto-load your CLAUDE.md
claude

# Try asking Claude about your project
> What conventions does this project use?

# Claude should reference your CLAUDE.md rules
Tip: Claude Code automatically discovers .claude/agents/, .claude/commands/, and .claude/skills/ folders. Any .md file you place in these folders becomes available immediately — no registration needed.

Try It Yourself

Create the .claude/ folder structure in your project, write a minimal CLAUDE.md, and start a Claude Code session. Ask Claude "What do you know about this project?" to verify it reads your config.

Hint

Start with just 10-20 lines in CLAUDE.md covering: project overview, how to build/test, and your top 3 conventions. You can expand later as you discover repeated corrections.

Key Takeaways

  • Claude Code installs as a global npm package
  • The .claude/ folder structure is convention-based — just create the folders
  • CLAUDE.md is auto-loaded every session — start small and grow
  • Agents, commands, and skills are just markdown files — no build step needed
TUTORIAL 03

CLAUDE.md — Your Project Brain

25 min Beginner

What You'll Learn

  • What CLAUDE.md is and why it's the most important file in your project
  • How to structure a production-grade CLAUDE.md
  • Real-world examples from different tech stacks
  • How other tools (Codex AGENTS.md, Cursor rules) solve the same problem

Context

CLAUDE.md is automatically loaded at the start of every Claude Code session. It tells Claude about your project, your conventions, your tech stack, and your rules. A well-crafted CLAUDE.md is the single highest-leverage thing you can do to improve Claude Code's output quality.

Walkthrough

Step 1: Anatomy of a Good CLAUDE.md

CLAUDE.md
# My SaaS App — Next.js + Supabase + Stripe

## Project Overview
A subscription management platform. Next.js 14 App Router,
Supabase for auth/DB, Stripe for payments.

## Critical Rules

### 1. Code Organization
- Many small files over few large files
- High cohesion, low coupling
- 200-400 lines typical, 800 max per file

### 2. Code Style
- Immutability always — never mutate objects or arrays
- No console.log in production code
- Proper error handling with try/catch
- Input validation with Zod

### 3. Testing
- TDD: Write tests first
- 80% minimum coverage
- Unit tests for utilities, integration for APIs

### 4. Security
- No hardcoded secrets
- Environment variables for sensitive data
- Parameterized queries only

## File Structure
src/
|-- app/              # Next.js app router
|-- components/       # Reusable UI components
|-- hooks/            # Custom React hooks
|-- lib/              # Utility libraries
|-- types/            # TypeScript definitions

## Build & Test
npm run dev          # Development server
npm test             # Run test suite
npm run build        # Production build
npm run lint         # ESLint check

## Key Patterns

### API Response Format
All API routes return this shape:
interface ApiResponse<T> {
  success: boolean
  data?: T
  error?: string
}

Step 2: What to Include

SectionPurposeExample
Project OverviewWhat this project does, tech stack"Next.js SaaS with Supabase + Stripe"
Critical RulesNon-negotiable conventions"Never mutate state", "TDD always"
File StructureWhere things liveDirectory tree with annotations
Build & TestHow to build, test, lintnpm test, 80% coverage
Key PatternsCode examples of your conventionsAPI response format, error handling
EnvironmentRequired env vars and setupRequired API keys, services

Step 3: CLAUDE.md Best Practices

  • Keep it under 100 lines — smaller context = faster, more focused responses
  • Show code examples of your conventions, don't just describe them
  • Include build/test commands — Claude needs to know how to verify its work
  • Update it when you find yourself repeating the same corrections
  • Use CLAUDE.md in subdirectories for folder-specific rules (they stack)
Cross-Tool Equivalents: In Codex, the same concept is AGENTS.md — loaded at session start with project instructions. In Cursor, you use .cursorrules files. The pattern is universal: tell the AI about your project upfront.

Try It Yourself

Create a CLAUDE.md for your own project. Include: project overview, critical rules, file structure, and how to run tests. Start a new Claude Code session and observe how it follows your conventions.

Hint

Start small. A 20-line CLAUDE.md that covers your top 5 rules is better than a 500-line document Claude can't parse. Add sections as you discover repeated corrections.

Key Takeaways

  • CLAUDE.md is automatically loaded every session — it's your highest-leverage config file
  • Cover: overview, rules, structure, build/test, key patterns
  • Show code examples of your conventions, don't just describe them
  • Same pattern exists in Codex (AGENTS.md) and Cursor (.cursorrules)
TUTORIAL 04

Agents — Your AI Team

30 min Intermediate

What You'll Learn

  • Agent anatomy: YAML frontmatter + markdown body
  • Key fields: name, description, model, tools, color, skills
  • Model routing: Opus vs Sonnet vs Haiku
  • Agent anti-patterns to avoid

Context

Instead of one AI doing everything, you create a team of specialists. A planner plans, a reviewer reviews, a tester writes tests. Each agent lives in .claude/agents/ as a markdown file with YAML frontmatter defining its identity, tools, and model tier.

Walkthrough

Step 1: Agent Anatomy

Every agent is a markdown file with YAML frontmatter:

.claude/agents/code-reviewer.md
---
name: code-reviewer
description: |
  Reviews code changes for quality, security, and adherence to
  project conventions. Use when reviewing PRs, after implementing
  features, or before merging code.

  Examples:
  - "Review the changes in the auth module"
  - "Check this PR for security issues"
  - "Review my implementation before I merge"
model: sonnet
tools:
  - Read
  - Grep
  - Glob
  - Bash
---

# IDENTITY

You are a code reviewer specializing in quality, security,
and convention adherence.

# STEPS

1. Read CLAUDE.md for project conventions
   - Load project rules, coding style, testing requirements
   - Note any project-specific patterns to check against

2. Gather changed files
   - Run `git diff --name-only` to find changed files
   - Filter to relevant source files (skip configs, locks)

3. Analyze code quality
   - Check each file for bugs and logic errors
   - Verify error handling completeness (try/catch, fallbacks)
   - Check function size (<50 lines), file size (<800 lines)
   - Look for mutation patterns, deep nesting, hardcoded values

4. Security audit
   - Flag OWASP Top 10: injection, XSS, broken auth, secrets
   - Check input validation on all user-facing functions
   - Verify parameterized queries for all DB calls
   - Scan for hardcoded API keys, tokens, passwords

5. Verify test coverage
   - Check that new code has corresponding tests
   - Verify edge cases are covered (null, empty, boundary)
   - Flag untested public functions

6. Generate report
   - List findings by severity: CRITICAL, HIGH, MEDIUM, LOW
   - Include: file, line, description, suggested fix
   - Summarize total findings and pass/fail recommendation

# RULES

ALWAYS_DO:
  - Decompose work into tasks with TaskCreate before starting
  - Mark each task in_progress/completed as you go
  - Read the project's CLAUDE.md first for conventions
  - Focus on substance over style
  - Report confidence level for each finding

NEVER_DO:
  - Modify any files (you are read-only)
  - Skip task tracking — every phase must be tracked
  - Report stylistic preferences unless they violate conventions
  - Flag issues in unchanged code unless CRITICAL

Step 2: Frontmatter Fields

FieldRequiredPurpose
nameYesKebab-case identifier (e.g., code-reviewer)
descriptionYesWhen to use this agent + 2-4 example triggers (50-500 chars)
modelYesModel tier: opus, sonnet, or haiku
toolsNoScoped tool list (Read, Write, Edit, Bash, Grep, Glob, Agent)
colorNoDisplay color: red, blue, green, yellow, cyan, magenta
skillsNoList of skill names this agent should load
memoryNoPath to persistent memory file

Step 3: Model Routing Strategy

Route tasks to the right model tier based on complexity:

Model Routing Guide
Haiku  ($)   — classification, boilerplate, narrow edits, fast tasks
Sonnet ($$)  — implementation, refactors, code review, testing
Opus   ($$$) — architecture, root-cause analysis, multi-file reasoning

Rule: Escalate model tier ONLY when lower tier fails
      with a clear reasoning gap.

Step 4: Tool Scoping as Security

Limit what each agent can do — this is a security pattern:

  • Planner: ["Read", "Grep", "Glob"] — read-only, cannot modify code
  • Reviewer: ["Read", "Grep", "Glob", "Bash"] — can run git diff, still no writes
  • Developer: ["Read", "Write", "Edit", "Bash", "Grep", "Glob"] — full access

Step 5: Agent Anti-Patterns

Common Agent Anti-Patterns:
  • Vague trigger phrases — "Use for general tasks" tells Claude nothing. Be specific with examples.
  • Over-permissioned tools — Giving Write/Edit to a reviewer agent defeats the purpose of separation.
  • No examples in description — Always include 2-4 concrete example prompts that trigger this agent.
  • Missing ALWAYS_DO/NEVER_DO — Without explicit rules, agents drift from their role.
  • God Agent — One agent doing everything. Decompose into focused specialists.
  • Skillless Agent — Every agent should reference at least one skill for domain knowledge.

Try It Yourself

Create a .claude/agents/planner.md agent with read-only tools (Read, Grep, Glob), the opus model, and a description with 3 example triggers. Give it ALWAYS_DO and NEVER_DO rules.

Hint

The planner should NEVER write code — only read and analyze. Its ALWAYS_DO should include: "Wait for user confirmation before any implementation begins." Its NEVER_DO should include: "Modify any files."

Key Takeaways

  • Agents are markdown files with YAML frontmatter in .claude/agents/
  • Key fields: name (kebab-case), description (with examples), model, tools
  • Tool scoping limits what each agent can do — security by design
  • Model routing (Opus/Sonnet/Haiku) balances quality vs cost
  • Avoid anti-patterns: vague triggers, over-permissioning, god agents
TUTORIAL 05

Commands — Slash Command Definitions

25 min Intermediate

What You'll Learn

  • How slash commands work in Claude Code
  • Command anatomy: name + description frontmatter + body
  • How commands dispatch work to agents via Task()
  • Building workflow chains: /plan → /tdd → /review

Context

Commands are the user-facing interface to your agent system. Instead of writing paragraphs explaining what you want, you type /plan and a specialist agent takes over. Commands live in .claude/commands/ as markdown files. The file name becomes the slash command: plan.md → /plan. User arguments are available as $ARGUMENTS.

Walkthrough

Step 1: Command Anatomy

Every command has YAML frontmatter with name and description, plus a markdown body:

.claude/commands/plan.md
---
name: plan
description: "Analyze requirements and create a step-by-step
  implementation plan. Usage: /plan Add auth to the API"
---

Analyze requirements and create a step-by-step implementation
plan for the requested feature.

Input: $ARGUMENTS

## FLOW
1. Read CLAUDE.md for project conventions
2. Scan relevant source files to understand current architecture
3. Restate the requirements in your own words
4. Identify risks and dependencies
5. Create a phased implementation plan
6. Present plan and WAIT for user confirmation

## RULES
ALWAYS_DO:
  - Break work into phases of 15-minute units
  - Identify the dominant risk for each phase
  - Wait for explicit "go ahead" before any implementation

NEVER_DO:
  - Write, edit, or create any files
  - Start implementation without user confirmation
  - Skip the risk assessment step

Step 2: Frontmatter Fields

FieldRequiredPurpose
nameYesThe slash command name (must match filename without .md)
descriptionYesShown in / autocomplete. Include usage example with arguments.

The body uses $ARGUMENTS to access whatever the user types after the command name. For example, /plan Add user auth makes $ARGUMENTS = "Add user auth".

Step 3: Commands That Dispatch to Agents

The most powerful pattern: commands that launch subagents via Task(). This delegates complex work to a specialized agent while the command orchestrates:

.claude/commands/write-test.md
---
name: write-test
description: "Write a new E2E test from a Jira test case.
  Usage: /write-test QA-0001"
---

Write a new E2E Playwright test from a Jira test case.

Input: $ARGUMENTS

Parse $ARGUMENTS to extract the test case ID (e.g., `QA-0001`).

Launch the subagent:

```
Task(
    description="Write and verify E2E test for $ARGUMENTS",
    subagent_type="e2e-test-writer",
    prompt="Write a new E2E test for test case $ARGUMENTS\n\n
Follow the full workflow:\n
1. Read test case from Jira\n
2. Understand what to test\n
3. Find similar tests in repo\n
4. Plan the test structure and present for approval\n
5. Write the test file, selectors, helpers\n
6. Run and verify interactively\n
7. Fix issues and burn test 3x\n
8. Sync test case back to Jira\n
9. Create branch and PR\n
10. Final report"
)
```
Key Pattern: The Task() call dispatches work to a subagent defined in .claude/agents/. The subagent_type matches the agent's name field. The prompt tells the agent exactly what to do. The command stays as the orchestrator — it delegates, it doesn't do the work itself.

Step 4: Simple Commands (No Agent Dispatch)

Not all commands need agents. Simple commands can work directly:

.claude/commands/review.md
---
name: review
description: "Review code changes for quality and security.
  Usage: /review"
---

Review all uncommitted code changes for quality, security,
and convention adherence.

## FLOW
1. Run `git diff --name-only` to find changed files
2. For each changed file, check for:
   - CRITICAL: Hardcoded secrets, SQL injection, XSS
   - HIGH: Missing error handling, large functions
   - MEDIUM: Missing tests, mutation patterns
3. Generate report with severity, location, description, fix
4. Block commit if CRITICAL or HIGH issues found

## RULES
NEVER_DO:
  - Skip the security check
  - Auto-fix issues without showing the report first

Step 5: The Golden Workflow

The most powerful pattern is chaining commands:

The Golden Workflow
# 1. PLAN — Understand the task before touching code
> /plan Add real-time notifications to the app

# 2. Implement (manually or via a /build command)

# 3. REVIEW — Review the implementation
> /review

The key insight: /plan never writes code. It creates a plan and waits for your approval. This prevents Claude from charging ahead with the wrong approach.

Step 6: Command Structure Best Practices

SectionPurpose
name (frontmatter)The /name used to invoke the command
description (frontmatter)Shown in autocomplete — include usage example with args
$ARGUMENTSUser input after the command name
Task()Dispatch work to a subagent — use for complex multi-step work
FLOWNumbered steps — strict order, no skipping
RULESALWAYS_DO / NEVER_DO guardrails
Command Anti-Patterns:
  • Missing name — Always include name: in frontmatter so the command is discoverable
  • No usage example in description — Include "Usage: /name arg" so users know the syntax
  • Inline complex logic — If a step needs 50+ lines, dispatch to a subagent with Task()
  • No phase structure — Commands without numbered FLOW steps produce unpredictable results
  • Missing edge cases — Handle: empty $ARGUMENTS, no files changed, agent failure
  • No final report — Every command should produce a visible output summary

Try It Yourself

Create a .claude/commands/test.md command with name: test and a description including usage. It should: (1) parse $ARGUMENTS for a test scope, (2) run the test suite, (3) report coverage, (4) suggest tests for uncovered code. Bonus: dispatch a test-runner agent via Task().

Hint

The frontmatter should be: name: test and description: "Run tests and report coverage. Usage: /test unit". For the Task() dispatch: Task(description="Run tests", subagent_type="test-runner", prompt="Run $ARGUMENTS tests and report coverage").

Key Takeaways

  • Commands live in .claude/commands/ — filename becomes the /command
  • Frontmatter requires name and description (with usage example)
  • $ARGUMENTS captures everything the user types after the command
  • Use Task() to dispatch complex work to subagents defined in .claude/agents/
  • Simple commands use FLOW steps directly; complex ones delegate via Task()
  • /plan NEVER writes code — it waits for your explicit approval
TUTORIAL 06

Skills — Domain Knowledge Packages

30 min Intermediate

What You'll Learn

  • What skills are and how they differ from CLAUDE.md rules
  • The full skill anatomy: SKILL.md, FORMAT.md, VOICE.md, workflows/, references/
  • Progressive disclosure: metadata → body → resources
  • How to create skills from your project's patterns

Context

CLAUDE.md rules are always-on constraints. Skills are contextual expertise — loaded when relevant. When working on API code, an api-design skill activates. When writing tests, a tdd-workflow skill provides guidance. Skills keep the context window lean by loading only what's needed.

Walkthrough

Step 1: Skill Anatomy

A skill is a folder with multiple files, each serving a specific purpose:

Skill Folder Structure
.claude/skills/git-commit/
├── SKILL.md           # Core knowledge (required)
├── FORMAT.md          # Output format specification (required)
├── VOICE.md           # Tone, terminology, report structure (required)
├── DRY_RUN.md         # Dry-run analysis behavior
├── workflows/         # Task playbooks per task type
│   ├── commit.md      # How to handle a normal commit
│   ├── amend.md       # How to handle amend requests
│   └── stage-review.md # How to review staging
├── references/        # On-demand documentation
│   └── conventional-commits.md
└── scripts/           # Deterministic bash scripts

Step 2: SKILL.md — The Core

.claude/skills/git-commit/SKILL.md
---
name: git-commit
description: |
  Create semantic git commits using Conventional Commits.
  Use when committing changes, reviewing staged files, or
  amending recent commits.
allowed-tools:
  - Bash
---

# IDENTITY
You create clean, semantic git commits following Conventional
Commits specification.

# WORKFLOWS MAP
- commit.md — Standard commit workflow
- amend.md — Amend a recent commit
- stage-review.md — Review what's staged before committing

# KNOWLEDGE

## PATTERNS
- Format: type(scope): description
- Types: feat, fix, refactor, docs, test, chore, ci, perf
- Use HEREDOC for multi-line commit messages
- Include Co-Authored-By when pair programming

## CONVENTIONS
- One logical change per commit
- Present tense imperative mood ("Add feature" not "Added")
- Never commit secrets or .env files
- Prefer staging specific files over `git add -A`
- Always create NEW commits after hook failures (never amend)

# RULES

NEVER_DO:
  - Commit files containing secrets
  - Amend without explicit user request
  - Skip pre-commit hooks (--no-verify)
  - Force push to main/master
  - Use git add -A without reviewing files

ALWAYS_DO:
  - Run git status before committing
  - Check git log for commit message style
  - Stage files by name, not with wildcards
  - Verify success with git status after commit

Step 3: FORMAT.md — Output Specification

.claude/skills/my-skill/FORMAT.md
# Output Format

## SECTION summary
FORMAT: prose
REQUIRED: true
> 2-3 sentence overview of what was done

## SECTION files
FORMAT: table
COLUMNS: File | Action | Description
REQUIRED: true

## SECTION quality_checklist
FORMAT: checklist
REQUIRED: true
- [ ] Tests pass
- [ ] No lint errors
- [ ] Security check passed

## SECTION issues
FORMAT: list
REQUIRED: false (only if issues found)
> Each issue: severity, location, description, fix

Step 4: VOICE.md — Communication Style

.claude/skills/my-skill/VOICE.md
# Voice

## TONE
- Direct, technical, no hedging
- Lead with action, not explanation
- Confidence without arrogance

## REPORTING
- Summary first, details on demand
- Use tables for structured data
- Bullet points for lists of findings

## TAGS
Inline issue markers for categorization:
- [BUILD] — Build or compilation issues
- [SECURITY] — Security vulnerabilities
- [PERF] — Performance concerns
- [TEST] — Test coverage gaps

## TERMINOLOGY
PREFER: "returns" over "gives back"
PREFER: "handles" over "takes care of"
PREFER: "validates" over "checks if valid"
AVOID: "simply", "just", "obviously"

Step 5: Progressive Disclosure

Skills use three disclosure tiers to minimize context usage:

TierWhen LoadedSize
Metadata (frontmatter)Always — used for skill matching~100 words
SKILL.md bodyWhen skill is triggered<5,000 words
references/, workflows/On-demand when neededUnlimited

Step 6: Skills vs CLAUDE.md Rules

AspectCLAUDE.md RulesSkills

Example"Never hardcode secrets""How to implement Django middleware"

Try It Yourself

Create a minimal skill at .claude/skills/my-first-skill/SKILL.md with frontmatter (name, description) and a KNOWLEDGE section documenting one pattern from your project. Add a FORMAT.md with at least a summary section.

Hint

Start with a pattern you repeat often. For example, if your project has a specific error handling pattern, document it in the KNOWLEDGE section with a code example. The description should mention when this skill activates.

Key Takeaways

  • Skills are multi-file knowledge packages in .claude/skills/*/
  • Full anatomy: SKILL.md + FORMAT.md + VOICE.md + workflows/ + references/
  • Progressive disclosure: metadata always, body when triggered, resources on-demand
  • Skills are contextual (on-demand), rules are universal (always-on)
  • Start with workflows, then derive knowledge — not the other way around
TUTORIAL 07

Hooks — Event-Driven Automation

35 min Intermediate

What You'll Learn

  • How hooks intercept Claude Code tool calls
  • Hook lifecycle: PreToolUse, PostToolUse, SessionStart, Stop
  • Real examples: blocking --no-verify, auto-formatting, quality gates
  • How to configure hooks in settings.json

Context

Hooks are the immune system of your development workflow. They fire automatically before or after Claude uses a tool, catching mistakes in real-time. A PreToolUse hook can block a dangerous action. A PostToolUse hook can auto-format code the instant it's written.

Walkthrough

Step 1: Hook Architecture

Every hook has three parts:

  • Matcher — Which tool(s) trigger it (Bash, Edit, Write, * for all)
  • Command — The script to run when triggered
  • Exit Code — 0 = pass/warn, 2 = block the action
.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "echo '$TOOL_INPUT' | grep -q '\\-\\-no-verify' && echo 'BLOCKED: --no-verify is not allowed' && exit 2 || exit 0"
        }],
        "description": "Block git hook-bypass flag"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{
          "type": "command",
          "command": "npx prettier --write \"$TOOL_INPUT_FILE_PATH\" 2>/dev/null || true"
        }],
        "description": "Auto-format after file edits"
      }
    ]
  }
}

Step 2: Blocking Dangerous Actions

How blocking works
Claude tries: git commit --no-verify -m "quick fix"
                          |
PreToolUse hook fires for "Bash" tool
                          |

Exit code 2 -> ACTION BLOCKED

Claude is forced to commit properly with hooks enabled

Common Hook Patterns

Step 2: Mode Detection

Agents should detect their operating mode from the user's input:

Mode Detection Example
# DETECT mode FROM input

- IF input mentions "review", "check", "audit"
  -> mode = review
  -> WORKFLOW: Read files, analyze, report findings

- IF input mentions "fix", "bug", "broken", "error"
  -> mode = fix
  -> WORKFLOW: Reproduce, diagnose, fix, verify

- IF input mentions "add", "create", "implement", "build"
  -> mode = feature
  -> WORKFLOW: Plan, implement, test, report

- IF input mentions "refactor", "improve", "clean up"
  -> mode = refactor
  -> WORKFLOW: Analyze, propose changes, implement, verify

- ELSE -> mode = default
  -> WORKFLOW: Analyze request, execute, report

Each MODE has its own TRIGGER, WORKFLOW, and TASKS — preventing the agent from using a one-size-fits-all approach.

Context

Tutorial 6 covered skill anatomy. Now we build one from scratch using the proven 6-step methodology. The key insight: start with workflows (day-to-day tasks), then derive knowledge — not the other way around.

Walkthrough

Step 1: Understand — Get Concrete Examples

Before writing anything, gather 3-5 concrete examples of the task this skill handles. Avoid abstract descriptions.

Good vs Bad Starting Points
BAD:  "A skill for API development"
GOOD: "When I add a new endpoint, I always:

Step 2: Plan Reusables

Identify what can be extracted into scripts, references, and templates:

  • Scripts — Deterministic bash operations (scaffold files, run checks)
  • References — Documentation loaded on-demand (API specs, conventions)

Step 3: Create the Folder Structure

Step 4: Write Workflows First

Workflows are the source of truth. Each workflow is a concrete playbook:

.claude/skills/api-development/workflows/feature.md
# Add New API Endpoint — Workflow

## IDENTITY
Playbook for adding a new API endpoint to the project.

## GUARD (prerequisites)
- [ ] Route path is defined (e.g., /api/v1/users)
- [ ] HTTP method is decided (GET, POST, PUT, DELETE)
- [ ] Request/response schema is clear
ON_FAIL: Ask the user to clarify before proceeding

## FLOW
1. Create route handler in `src/routes/`
   WITH: route path, HTTP method
   PRODUCING: handler file with input validation

2. Add Zod schema for request validation
   WITH: request body/query schema
   PRODUCING: schema file in `src/schemas/`

3. Write integration tests
   WITH: happy path + 3 error cases
   PRODUCING: test file with 4+ test cases

4. Update OpenAPI spec
   WITH: endpoint definition
   PRODUCING: updated openapi.yaml

5. Add rate limiting middleware
   WITH: default 100 req/min
   PRODUCING: middleware chain

## CHECKLIST (verification)
- [ ] Handler returns correct status codes
- [ ] Input validation rejects invalid data
- [ ] Tests cover happy path and error cases
- [ ] OpenAPI spec matches implementation
- [ ] Rate limiting is configured

Step 5: Derive SKILL.md from Workflows

Now write the SKILL.md by extracting patterns from your workflows:

.claude/skills/api-development/SKILL.md
---
name: api-development

  API endpoint development patterns. Use when creating new
  endpoints, fixing API bugs, or reviewing API code.
---

# IDENTITY
You follow this project's API development conventions.

# WORKFLOWS MAP (source of truth)
- feature.md — Add a new endpoint
- bugfix.md — Fix an API bug
- review.md — Review API changes

# KNOWLEDGE (derived from workflows)

## PATTERNS
- All endpoints use Zod for input validation
- Response format: { success, data?, error? }
- Rate limiting: 100 req/min default, configurable per route
- OpenAPI spec must match implementation

## CONVENTIONS
- Route handlers in src/routes/
- Schemas in src/schemas/
- Tests alongside code (*.test.ts)
- Always return proper HTTP status codes

## STRATEGY
CAPTURE: New patterns discovered during reviews
UPDATE_FREE: Add new patterns to reference docs
UPDATE_APPROVAL: Changes to validation or response format

Step 6: Skill Anti-Patterns

Skill Anti-Patterns:
  • Thin Skill — Only a SKILL.md with no workflows or references. Use the full folder structure.
  • Abstract Patterns — Knowledge not derived from actual workflows. Delete unused patterns.
  • Empty References — Reference folder exists but files are empty. Seed with real docs.
  • Placeholder Workflows — "TODO: add steps." Rewrite with concrete actions.
  • Bloated SKILL.md — Over 5,000 words. Move details to references/.
  • Dead References — Reference files that no workflow ever loads. Remove them.

  • 6 steps: Understand, Plan reusables, Create structure, Write workflows, Derive SKILL.md, Iterate

Context

Code review shouldn't be optional — it should be automated. By creating a code-reviewer agent and a /review command, you can run security and quality checks on every change. The agent hunts for vulnerabilities while the command orchestrates the workflow.

Walkthrough

Step 1: Security Review Checklist

Your security reviewer agent should check the OWASP Top 10:

Security Review Patterns
## OWASP Top 10 Check
1. **Injection** — Queries parameterized? Input sanitized?
2. **Broken Auth** — Passwords hashed (bcrypt/argon2)?
3. **Sensitive Data** — HTTPS enforced? Secrets in env vars?
4. **XXE** — XML parsers configured securely?
5. **Broken Access** — Auth checked on every route?
6. **Misconfiguration** — Debug mode off in prod?
7. **XSS** — Output escaped? CSP set?
8. **Insecure Deserialization** — Safe deserialization?
9. **Known Vulnerabilities** — Dependencies up to date?
10. **Insufficient Logging** — Security events logged?

Step 2: Critical Code Patterns

Patterns That Must Be Flagged
// CRITICAL: SQL injection via string concatenation
const query = `SELECT * FROM users WHERE id = ${userId}`;
// FIX: Use parameterized queries
const query = `SELECT * FROM users WHERE id = $1`;
const result = await db.query(query, [userId]);

// CRITICAL: Hardcoded secrets
const API_KEY = "sk-abc123...";
// FIX: Use environment variables
const API_KEY = process.env.API_KEY;

// HIGH: XSS via innerHTML
element.innerHTML = userInput;
// FIX: Use textContent or DOMPurify
element.textContent = userInput;

// CRITICAL: No auth check on route
app.post('/admin/delete-user', deleteUser);
// FIX: Add authentication middleware
app.post('/admin/delete-user', requireAuth, requireAdmin, deleteUser);

Step 3: Confidence-Based Filtering

Reduce noise by only reporting findings above a confidence threshold:

Reviewer Rules
## Confidence-Based Filtering

**Report** if >80% confident it is a real issue
**Skip** stylistic preferences unless they violate project conventions
**Skip** issues in unchanged code unless CRITICAL security issues
**Consolidate** similar issues ("5 functions missing error handling")
**Prioritize** issues that could cause bugs, security vulns, or data loss

Step 4: Review Severity Levels

SeverityActionExamples
CRITICALBlock commit, fix immediatelySQL injection, hardcoded secrets, XSS
HIGHFix before mergeMissing auth, large functions, no error handling
MEDIUMAddress in next iterationMissing tests, mutation patterns
LOWTrack for laterStyle suggestions, minor improvements

Key Takeaways

  • Automate security review with a dedicated agent and /review command

Orchestration — Multi-Agent Pipelines

35 min Advanced

What You'll Learn

  • How to chain agents into multi-phase pipelines
  • Pipeline YAML definitions with phases and gates
  • Context contracts between pipeline phases
  • The orchestrator command pattern (dispatcher + workers)

Context

Individual agents handle one task well. But real features require a pipeline — planning, then architecture, then implementation, then testing, then review. Multi-agent orchestration chains these specialists into a repeatable process with quality gates between each phase.

Walkthrough

Step 1: Pipeline YAML Definition

.claude/workflows/my-pipeline.yaml
name: feature-pipeline
description: Full feature development pipeline

phases:
  - name: requirements
    agent: pm-agent
    mode: feature
    description: Write requirements and acceptance criteria
    input: user_request
    output: prd_document
    validation:
      - PRD document exists
      - Acceptance criteria defined

  - name: architecture
    agent: architect-agent
    mode: design
    description: Design package structure and interfaces
    input: prd_document
    output: architecture_design
    depends_on: [requirements]
    validation:
      - Interfaces defined
      - No circular dependencies

  - name: implementation
    agent: dev-agent
    mode: feature
    description: Implement the feature
    input: architecture_design
    output: implementation_result
    depends_on: [architecture]
    validation:
      - Code compiles
      - Tests pass

  - name: review
    agent: reviewer-agent
    mode: review
    description: Review for quality and security
    input: implementation_result
    output: review_report
    depends_on: [implementation]
    validation:
      - No CRITICAL issues
      - No HIGH security findings

error_handling:
  on_phase_failure: stop_pipeline
  on_validation_failure: retry_phase_once

context_contracts:
  requirements_to_architecture:
    detail_level: full
    max_lines: 500
  architecture_to_implementation:
    detail_level: sections
    max_lines: 300
  implementation_to_review:
    detail_level: file_manifest
    max_lines: 200

Step 2: Context Contracts

Control how much context passes between phases to avoid the "Context Snowball" anti-pattern:

Detail LevelWhat's PassedUse When
fullComplete outputNext phase needs everything (rare)
summaryKey findings onlyOverview is sufficient
sectionsSpecific named sectionsOnly certain parts needed
file_manifestList of files changedReviewer just needs file list
omitNothingPhase is independent

Step 3: The Orchestrator Command

The orchestrator command reads the pipeline YAML and dispatches agents:

Orchestrator Pattern
Orchestrator Command (main thread)
       |
       +---> Phase 1: PM Agent -> PRD document
       |         (validation gate)
       +---> Phase 2: Architect Agent -> Design
       |         (validation gate)
       +---> Phase 3: Dev Agent -> Implementation
       |         (validation gate)
       +---> Phase 4: Reviewer Agent -> Report
       |
       Done: Summary of all phases

Step 4: Pipeline Design Rules

  • PM agent is always Phase 0 — Requirements before everything
  • Star topology — Only the orchestrator dispatches workers. Workers never talk to each other.
  • Anti-do-it-yourself — The orchestrator NEVER does workers' jobs. It dispatches and collects.
  • Context contracts — Every phase-to-phase handoff has explicit max_lines and detail_level.
  • Validation gates — Each phase must pass validation before the next starts.
Pipeline Anti-Patterns:
  • Context Snowball — Later phases get ALL upstream output. Use context contracts with max_lines.
  • Self-Doing Dispatcher — Orchestrator does workers' jobs instead of delegating.
  • Missing PM — No requirements phase. Always start with a PM agent.
  • Blind Handoff — No context schema between phases. Add explicit contracts.
  • Setup as Phase — One-time scaffolding as a pipeline phase. Make it a standalone command instead.

Key Takeaways

  • Pipelines chain agents into phased workflows with validation gates
  • Context contracts control what passes between phases (prevent context snowball)
  • Star topology: orchestrator dispatches, workers report back, no worker-to-worker
  • PM agent is always Phase 0 — requirements before implementation
